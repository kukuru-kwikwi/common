# General

## Localization

- All user-facing text must support localization. Use resource files for strings and ensure UI components can display translated content.
- Date, time, and number formats should adapt to user locale.
- data-models resources should be under the same data-model resource file
  - Field related should respect the format : {model}.{field}
    - label should repsect the format : {model}.{field}.{label}
    - warning should repsect the format : {model}.{field}.{warning}
    - error should repsect the format : {model}.{field}.{error}
  - model related
    - message should respect the format : {model}.{messages}.{key}
    - warning should respect the format : {model}.{warnings}.{key}
    - error should respect the format : {model}.{errors}.{key}
- Form and other UI related resources should be under a separate file

## Audits

- Every entity (e.g., bike, assignment) maintains a comprehensive, immutable audit/history log.
- The system audit log records all changes: creation, updates, deletions, and significant actions, including who performed the action and when.
- Audit entries include metadata: user ID, timestamp, action type, and details of the modification.
- The audit log is never modified or deleted; it is append-only for full traceability and debugging.
- Latest audit informations are summed into the standard audit fields
  - createdAt
  - createdBy
  - updatedAt
  - updatedBy
  - deletedAt
  - deletedBy
- Curated audit views (e.g., for mechanics) are generated by filtering and anonymizing entries from the system log, not by duplicating data.
- Use metadata (type/category) to support filtering for significant events (e.g., maintenance, part replacement).
- User identity may be anonymized for privacy in curated views.
- Permissions control access to audit views: owners see all history, mechanics see only significant, anonymized events.
- This approach avoids data duplication, ensures consistency, and supports flexible, permission-based history presentation.

## Validation

- Forms
  - All fields should be validated upon form load
  - All fields with errors should be flagged
  - All fields with errors should display their respective errors
  - Error summary should be displayed at the end of the form

## Error Handling

- Standardize error responses across all APIs and UI components.
- Use clear, actionable error messages for users.
- Log errors with sufficient context for troubleshooting.

## Logging

- Log key actions, changes, and errors for audit and debugging purposes.
- Log at the API level.
  - Every action on the api should be logged
  - Errors should have a unique id for tracability
  - Errors detail should be added to the log if available
- Follow privacy guidelines and avoid logging sensitive data.

## Telemetry

- Use OpenTelemetry standards and tooling to instrument the application and collect telemetry data (metrics, traces, logs).
- Export metrics data from OpenTelemetry to a Prometheus server for storage and querying.
- Use Grafana as the visualization frontend, connected to Prometheus, to create dashboards and monitor system health.
- Collect usage metrics to improve features and monitor system health.
- Ensure telemetry respects user privacy and opt-out settings.

## Security

- Enforce authentication and authorization for all operations.
- Validate all inputs and protect against common vulnerabilities (e.g., injection, XSS).
- Reference security requirements in each feature spec for operation-specific details.

## Data tables

- Data tables should use [tanstack table](https://tanstack.com/table/v7)
- Columns should be filtrable and orderable
- pagination should be enabled by default
- global search should be enabled by default
